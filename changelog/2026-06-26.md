# 2026-06-26 抗侧推力迈步恢复（失衡门控 + 推力课程，续训）

文件：`oceanisaaclab_env.py` / `oceanisaaclab_env_cfg.py` / `agents/rsl_rl_ppo_cfg.py`

## 目标与认识
- 从 06-25 稳定站立模型 `bdx_stand_stable/2026-06-25_20-03-14/model_2999` 续训，使其抗侧推力。
- 小推力靠踝/髋调节，**大推力物理上必须迈步**重建支撑面。站立版禁止迈步（`stand_still=-2.0`/`feet_air_time=0`）故扛不住大推力。

## 失衡门控（迈步是手段不是目的）
判据选择（sim2real 关键）：
- ❌ 特权推力信号（真机无、窗口对不上）；❌ base 线速度（IMU 积分漂移、四元数解不出线速度，sim 的 `root_lin_vel_b` 是特权）。
- ✅ `tilt=|proj_g_xy|` + `tilt_rate=|ang_vel_xy|`：IMU 可靠量、已在观测里。

实现（`_get_rewards`/`compute_rewards`）：
- `instability = sigmoid(sharpness·[(tilt-θt)/θt + (tilt_rate-θr)/θr])` ∈[0,1]。
- `feet_air_time` × **instability**（仅失衡奖励迈步）；`stand_still`/`feet_slide` × **(1-instability)**（稳态钉地，失衡放行）；靠 upright/alive 驱动恢复。
- cfg：`instability_tilt_thresh=0.15`、`instability_tilt_rate_thresh=1.0`、`instability_sharpness=8.0`、`rew_scale_feet_air_time` 0→1.0。

## 续训复盘：不迈步的根因是推力课程步数算错（量级 bug）

`bdx_stand_stable/2026-06-26_10-11-23`（4096env×8000iter）训完后 sim2sim：model_7999 会摔，model_4000 能站稳，但拖动时几乎不抬腿，只在快摔时才急摆两下腿。诊断结论：**不是奖励权重问题，是推力课程从未真正爬到大推力档。**

### 根因
`_current_push_force_range()` 的课程进度：
```python
frac = min(1.0, self.common_step_counter / float(self.cfg.push_curriculum_steps))
```
`common_step_counter` **每个控制步 +1，不乘 num_envs**（见 `direct_rl_env.py` 的 `self.common_step_counter += 1`）。所以 1 iter = `num_steps_per_env=24` 步：
- 训到 7999 iter，计数器才 `7999×24 ≈ 192,000`。
- 旧 `push_curriculum_steps = 250_000_000` → `frac ≈ 0.0008`，全程≈0。
- 推力范围整训卡在起点 **(20,40)N**，从没爬到终点 (45,90)N。

旧值是误按"乘了 num_envs(4096)"去估的（旧注释 `≈4096env×24×2500iter` 正暴露此错），正好差一个 4096 倍。

### 为何完美解释现象
- 小推力靠踝/髋调节即可恢复，**物理上不需要迈步** → 策略学到的最优解就是绝不迈步（正确行为，只是推力不够大）。
- 失衡门控 `instability` 接对了：只有快摔（tilt/tilt_rate 高）时才解锁迈步奖励、解除钉地惩罚 → 所以"只在快摔时甩两下腿"。
- 曲线印证：`feet_air_time` 全程负且平（4000→-0.030 / 7990→-0.034，脚基本不离地）；`stand_still` 惩罚一路涨（4000→-0.63 / 7990→-0.89，越训越钉死）；`fall` 反升（4000→0.125 / 7990→0.375）。**7999 比 4000 更易摔 = 后段过度钉地、刚性过强，对 sim2sim 动力学差异脆。4000 是甜点。**

### 修改
- `push_curriculum_steps`：`250_000_000` → **`60_000`**（≈2500 iter 跑满课程，= 跑满 iter 数 × 24）。注释补上"`common_step_counter` 不乘 num_envs"的坑。
- 奖励权重结构不动——问题在课程没触发，不在配比。

## 课程修复后复盘：迈步奖励被 air_time_target 反噬（仍不迈步）

课程修好后新 run `bdx_stand_stable/2026-06-26_14-30-44`（从 model_4000 续训到 8999）。play 不摔、sim2sim 能站住，但**关节非零位、拖动仍不迈步**。结合训练曲线 + sim2sim 数据（`oceanbdx/runlog/sim2sim_20260626_15*.csv`）定位到两件事：

### 现象一：关节非零位 = 正常（不用追）
两次 sim2sim 关节稳定偏离 0 约 0.1–0.3 rad（最大~17°），base_z≈0.40（略低于 0.42）。这是 RL 学出的微蹲撑姿：`rew_scale_joint_pos=-0.08` 相对 `upright=2.0`+`height=1.0` 太弱，策略停在比零位更稳的姿态。通常更稳、sim2real 无碍，不建议刻意追零位。真要贴零位 → 加大 `rew_scale_joint_pos`（如 -0.3），代价是牺牲稳定性。

### 现象二：不迈步的新根因 = `air_time_target=0.4` 反噬迈步
课程**确实生效了**（推力真爬到大档），证据：`fall` 终止从 4001 的 0 飙到 6000 的 10.6、8999 的 17.7（撑不住该迈步了）。但同期 `feet_air_time` 奖励**全程为负**（-0.007→-0.21→-0.13）。

矛盾根因：`feet_air_time = (air_time − air_time_target) × first_contact × instability`，而推力恢复是**急促碎步（腾空~0.1–0.25s）全 < 0.4** → 每次落地 `(air_time−0.4)<0` → **迈步反被扣分**。失衡门控本身是对的（sim2sim 拖动数据实测：失衡>0.5 时腿速 1.81 vs 稳态 0.10，差 18×，钉地惩罚确被解除），但 `feet_air_time` 这一项在惩罚真正的迈步动作 → 策略最优解变成"失衡时用踝/髋顶住、顶不住就摔"。

### 修改
- `air_time_target`：`0.4` → **`0.15`**（匹配急促恢复步，使恢复步变净正奖励）。
- `rew_scale_feet_air_time`：`1.0` → **`2.0`**（盖过迈步时产生的 joint_vel/action_rate/contact_force_rate 平滑惩罚）。
- 课程步数 `60_000` 不动（已验证生效）。

