## 方向查证结论（静态 URDF 分析，已可定性；PC 端 diag 再最终确认）
逐条证据全部指向同一结论：**URDF base_link 的 +x 指向机器人尾部，头部朝 −x**。
1. `neck_n1_joint`（颈/头链根）挂在 base_link **x = −0.074**（头在 −x 侧）；腿挂在 x = +0.021。
2. `leg_r1_joint` 在 **y = +0.045**、`leg_l1_joint` 在 y = −0.047：面朝 −x 时右手边恰为 +y，命名自洽。
3. sim2sim 实测「+vx 命令朝物理后方走得好」与上面完全一致：策略确实学的就是 body +x（= 物理尾部方向）。

由此昨天所有现象自洽解释：
- 训练里 vx 命令恒为正（0.10~0.25），只训练了「朝 body +x = 尾部方向」走 → W 走得好但方向是物理后方。

- E 旋转最好：yaw 与前向定义无关，不受此 bug 影响，所以是学得最正的一项。

## 具体改动（最小侵入：只改前向定义，不动观测坐标系）
修复层选择：不改 URDF（会牵连所有关节系/mesh）、不改 sim2sim 键位（治标）；在 env 奖励层重定义「前向」。

### cfg
- 新增 `forward_vx_sign = -1.0`：head-forward speed = `forward_vx_sign * root_lin_vel_b[:, 0]`。注释里写明 URDF 证据。观测不变（仍是 base 系原始量，sim/real IMU 一致），只有奖励和 reset 注入用它。

### env
- `compute_rewards` 新增参数 `forward_vx_sign`；`vx_act = forward_vx_sign * root_lin_vel_b[:,0]`，`track_lin_vel` / `progress` / `fwd_gate` 全部改用头部前向速度。
- `_reset_idx` 的 reference-state 初速度注入：`default_root_vel[:,0] = forward_vx_sign * vx_seed`（reset 后 yaw≈0，world x ≈ body x，头朝 world −x）。
- 其余项（lateral 用 vy²、upright/instability 用范数）对 180° 翻转不敏感，无需改。

### 诊断脚本
- `diag_walk.py`：前向速度按 `cfg.forward_vx_sign` 校正后报告（否则修正后跟踪比会显示成负的）。
- 新增 `diag_direction.py`：①量 neck_n4_link 在 base 系下 x 坐标（应为负）；②给 body+x 初速度看位移方向；打印 forward_vx_sign 应取的值。

## 追加：前后双向行走（vx 命令支持负值）

背景：方向符号修正后，训练命令 vx 仍恒为正（只训练「朝头部前向」单向走），S/后退方向仍在分布外。用户要求前后都能走，故把 vx 命令扩成双向。

### 改动（最小侵入，不动课程/奖励 scale，只让负 vx 进入分布 + 修方向相关的数学）

文件：`oceanisaaclab_env_cfg.py` / `oceanisaaclab_env.py`

- **cfg**：新增 `backward_prob = 0.5`——实际平移(非站立/非原地转向)样本中，把 vx 命令翻转为负(后退)的比例。命令幅度课程 `command_vx_range(_start)` 仍是正的幅度范围，翻转在采样后按概率施加，所以前后共享同一套速度课程，对称放开。
- **`_reset_idx`**：采样正 vx 幅度后，对 `moving = non_standing & ~turn_in_place` 的子集按 `backward_prob` 翻转符号 `*= -1.0`。reference-state 初速度注入的 `moving` 定义上移复用（原本在下方重复定义，已合并），`vx_seed` 现携带命令符号→后退命令注入后退初速度。
- **`compute_rewards` 方向门控泛化（关键，否则负命令数学会崩）**：
  - 旧 `fwd_frac = clamp(vx_act / clamp(vx_cmd, min=1e-3), 0, 1)`：对负 vx_cmd，`min=1e-3` 的 epsilon 保护失效、分母符号也错。
  - 新：`vx_cmd_sign=sign(vx_cmd)`、`vx_cmd_abs=|vx_cmd|`；`fwd_frac = clamp((vx_act*vx_cmd_sign)/clamp(vx_cmd_abs,min=1e-3), 0, 1)`。`vx_act*vx_cmd_sign` 把实际速度投影到命令方向，前/后走对了都为正、走反了 clamp 到 0。`forward_command = (vx_cmd_abs > move_command_threshold)`。
  - `progress`/`fwd_gate`/`progress_pay` 全部随之变成「命令方向的进度」，语义对前后对称。
  - `track_lin_vel`（`(cmd-vx_act)²`）与 `command_norm`（用 abs）本就对称，无需改。

## 追加：零命令站不住一直踏步——关闭 instability gate（奖励层根因）

背景：关推力版训到 ~5000 iter sim2sim 实测：前后左右方向都对了、给命令反应比带推力时稳一点，但**零命令(command=0)仍踏步不能钉地站住**，整体还有些晃。截图 TB：`single_support`≈26、`phase_contact`≈47、`feet_air_time`≈0.95 都很高（策略"很爱迈步"），`Loss/value` 50~220 剧烈抖（return 不平滑，与"总在晃"一致）。

### 根因（奖励门控漏洞，零命令 moving_command=0 / stand_command=1 时）
`instability = sigmoid(...)` 本为"被推失衡时解锁迈步恢复"设计，但 `enable_random_push=False` 后失衡只来自策略自身抖动，两处 gate 被"自己晃"反向利用：
1. `stable_stand_gate = stand_command * stability`（`stability=1-instability`）：踏步→晃→instability↑→stability↓→**钉脚 `stand_still` / 抗抖 `contact_force_rate` / 平滑 `motion_smooth` 惩罚全被削弱**。
2. `stepping_gate = max(moving_command, instability) = instability`：踏步→晃→instability↑→**`feet_air_time` 迈步腾空反被奖励**。

合力 = 零命令下"晃一晃/踏步"既免惩罚又拿奖励 → 踏步成了零命令的稳定漏洞。这是奖励层根因，不是训练不够。

### 改动（`oceanisaaclab_env_cfg.py` / `oceanisaaclab_env.py`，单变量、开关化、面向未来）
- cfg 新增 `enable_instability_gate = False`。
- env `_get_rewards`：算完 `instability` 后，若开关关闭则 `instability = zeros_like(instability)`。一处置零即让全部下游 gate 归位：
  - `stable_stand_gate → stand_command`（零命令**硬钉脚**，不能靠晃逃避）
  - `stepping_gate → moving_command`（零命令**不再奖励迈步**）
  - `progress_pay → fwd_gate`、`contact_smooth/motion_smooth` 零命令拉满（满抗抖/满平滑）
- **关键：对有命令(行走/转向)的 env 零影响**——`moving_command=1` 时 `stepping_gate=max(1,0)=1`、`progress_pay=fwd_gate`、`stable_stand_gate=0` 全与旧版一致。改动精准只收紧零命令站立，不动已学会的前后左右行走。
- 恢复带推力训练时**必须改回 `True`**（抗推需要失衡解锁迈步恢复），否则被推时钉脚救不回来会摔。

### 冒烟验证（train 64 env × 3 iter）
无 NaN/shape 报错；`stand_still` 出现非零负值（硬钉脚生效，脚一动即罚），其余项正常。

## 追加：完全关 gate 训练成功但 sim2sim 一踏步就摔——重开 gate 但调迟钝阈值

背景：关 gate 版训到 18700 iter，analyze_tb（run `2026-07-02_18-16-14`）显示**训练本身非常成功**：`fall` 3.5→**0.23**（几乎不摔）、`mean_episode_length`≈**476**、`Loss/value`≈**110 稳定**（上一版 50~220 抖，这版明显更稳）、`Policy/mean_std`≈**0.081** 收敛、`stand_still`≈-0.15（硬钉脚生效）。但 sim2sim **进 RL 一踏步就摔**。

### 根因（训练好 + sim2sim 摔 = gap × 无恢复能力）
完全关 gate（`instability≡0`）= 零命令**完全无迈步恢复能力**。训练分布内很干净所以不摔（fall=0.23）；但 sim2sim 进 RL 瞬间有姿态/动力学 gap 扰动，策略被扰到倾斜时因"零命令绝不迈步"的硬约束只能硬撑→救不回来→摔。**失衡迈步恢复是必要的鲁棒性，不能全关**。先前"零命令踏步"的真正病因不是 gate 本身，而是**阈值太敏感**：`tilt_thresh=0.15`≈8.6°，正常动态调整躯干晃这么多就触发解锁踏步。

### 改动（`oceanisaaclab_env_cfg.py`，只调 gate 阈值，代码逻辑不变）
- `enable_instability_gate` False→**True**（重新启用失衡迈步恢复）。
- `instability_tilt_thresh` 0.15→**0.28**（≈16°，明显前/侧倾才算失衡）。
- `instability_tilt_rate_thresh` 1.0→**2.5**（只有快速倾倒才算）。
- `instability_sharpness` 8.0 保持。
- 数值验算：稳态/踏步微晃(tilt 0.05~0.1, rate 0.2~1.0)→gate_arg 深负→`instability`≈0→硬钉脚不踏步；真失衡(tilt 0.3≈17°, rate 3.0)→`instability`≈0.89→解锁迈步救回。兼顾"零命令不踏步"+"真失衡能恢复不摔"。

### 冒烟（train 64 env × 3 iter）
无报错；`stand_still` 有非零值（微晃时 instability≈0，硬钉脚保留）。

### 训练与验证
