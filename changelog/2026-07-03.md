# 2026-07-03 对照迪士尼 BDX 方案的最小修正集（路线 A）

文件：`oceanisaaclab_env.py` / `oceanisaaclab_env_cfg.py` / `scripts/diag_walk.py`

## 背景

与迪士尼 BDX（Open Duck / AWD go_bdx 复刻）逐项对照后的诊断结论：本工程 21 项手工奖励
的纯塑形范式与 BDX 的「参考轨迹模仿 + 速度命令跟踪」范式存在结构差距，但在重构为模仿
范式（路线 B，中期做）之前，有 5 个确定配错的地方可以在现框架内修正（路线 A）。
按优先级实施如下。**观测/奖励语义均有变化，必须从头重训，旧 checkpoint 全部作废。**

## 1. 侧向 vy 命令训练（左摔的直接原因）

- 此前 `_commands[:,1]` 从未采样恒为 0，且 `rew_lateral=-0.8` 无条件惩罚一切 vy²。
  sim2sim 左/右平移按键对策略是**完全分布外输入 + 训练中被惩罚的行为**。
- 本机无踝滚转关节（r1 髋 yaw、r2 髋 roll、r3 髋 pitch、r4 膝、r5 踝 pitch），侧向稳定
  只能靠髋 roll + 迈步，不训 vy 等于侧向稳定能力整体缺失。
- 改动：
  - cfg 新增 `command_vy_range = (-0.15, 0.15)`（BDX mini ±0.2 / go_bdx ±0.3，保守起步）。
  - `_reset_idx` 对所有非站立 env 采样 vy（|vy|<阈值钳 0）；参考态初速度注入同步带 vy。
  - `track_lin_vel` 改为 (vx, vy) 平面 2 维跟踪；`rew_lateral` 改为惩罚 `(cmd_vy - vy_act)²`
    偏差而非 vy 本身。vy 的头部系约定与 vx 一致（head-left = forward_vx_sign * body_y，
    180° yaw 翻转对 x/y 同时变号）。
  - `fwd_gate` / `progress` 从 vx 标量推广为平面向量投影：实际平面速度在命令方向上的
    投影比例，语义不变（原地踏步拿 0、达到命令速度拿 1），侧向命令同样受益。

## 2. 零命令冻结 gait clock（零速踏步的结构性原因）

- 此前 `_gait_phase()` 基于 `episode_length_buf` 恒定走表，零速命令下观测里的 gait clock
  仍在 0.6s 周期振荡：策略一边被 stand_still 罚踏步、一边持续看到"该迈步了"的时钟信号。
- 改动：`_get_observations` 中零速命令时 gait clock 置 (0,0)——一个与任何行走相位都可
  区分的"站立"信号（对照 BDX 零命令钳制到站立参考的做法）。

## 3. upright 项加入行走前倾标称姿态（行走姿态差的直接配置错误）

- BDX 的躯干前倾是参考步态硬约束（go_bdx 约 8°、Open Duck mini 约 3°）：无踝滚转、踝
  只有 pitch 的腿型，行走时必须把重心压到支撑面前部。旧 upright 项无条件奖励绝对竖直，
  在**主动对抗**行走所需的自然前倾。
- 改动：cfg 新增 `walk_lean_angle = 0.087`（≈5°，从 3°~8° 中值试起）。移动命令下 upright
  目标改为 `proj_g_x = forward_vx_sign * sin(lean)`（头部朝向低头前倾），站立命令仍竖直。
  倾角远低于 instability gate 阈值（0.28≈16°）与摔倒判定，无联动风险。

## 4. 领域随机化（sim2sim 一碰就摔的主因候选）

- "训练 fall=0.23 但 sim2sim 一踏步就摔"是 DR 不足的典型症状：此前只有观测噪声 + 动作
  延迟。BDX 复刻还随机化质量/摩擦/PD 增益/重力/初始姿态。
- 改动：cfg 新增 `enable_domain_rand=True` + 三个幅度区间；env 新增
  `_apply_domain_randomization()`（startup 每 env 采样一次固定参数，数千 env 覆盖分布）：
  - 质量：全 link × U(0.8, 1.2)（BDX ×[0.5,1.5]，先保守，验证后再放大）。
  - PD 增益：各执行器 stiffness/damping 独立 × U(0.8, 1.2)，implicit actuator 同步写回 sim。
  - 摩擦：碰撞体 static/dynamic friction × U(0.7, 1.3)（对照 BDX），经 `root_physx_view`；
    后端不暴露该 view 时告警跳过。

## 5. 推扰重开：间隔 5~10s（晃悠姿态的诱因）

- 07-02 曾全关推力隔离验证。对照结论：问题不在推力本身而在密度——旧间隔 1~2s 使 8s
  episode 里机器人永远处于"刚被推/即将被推"状态，学出"常态微失衡随时准备迈步"的晃悠
  姿态是必然（Open Duck 推扰是每 5~10s 一次）。
- 改动：`enable_random_push=True`，`push_interval_s=(5.0, 10.0)`，力度沿用 (8,20)N 起步
  + 原课程爬到 (20,40)N。无扰稳定站走成为占绝对主导的训练状态，推扰只是偶发事件。

## 6. 低速段 + 双脚接触观测

- `command_vx_range` 从 (0.10, 0.25) 改为 (0.0, 0.25)（课程起点 (0.0, 0.18)），补上
  0~0.10 低速微调命令空洞；采样后 |vx|<move_command_threshold 钳 0，站立/移动语义清晰。
- 观测加入双脚接触布尔量（真机脚底开关可得；BDX 观测集也有），对步态相位对齐帮助大。
  **observation_space 41 → 43**（顺序：ang_vel 3 + proj_g 3 + cmd 3 + gait_clock 2 +
  feet_contact 2 + Δjoint_pos 10 + joint_vel 10 + last_action 10）。

## Bugfix：训练启动即崩——DR 摩擦随机化的 warp/torch API 误用

现象：`train.py` 一启动就在 env `__init__` 报错崩溃：
```
File ".../oceanisaaclab_env.py", line 142, in _apply_domain_randomization
    friction_scale = sample_uniform(lo, hi, (materials.shape[0], 1, 1), materials.device)
TypeError: rand() received an invalid combination of arguments - got (int, int, int, device=Device)
```

根因（`_apply_domain_randomization` 第 3 步摩擦随机化，两处 API 用法错）：
1. **`physx_view.get_material_properties()` 返回的是 warp 数组而非 torch 张量**，其 `.device`
   是 warp `Device` 对象。`sample_uniform` 内部 `torch.rand(*size, device=device)` 收到 warp
   Device 就报 `invalid combination of arguments`。质量/PD 两步用 `self.device`（合法 torch
   设备）所以没事，唯独摩擦这步误用了 `materials.device`。
2. **写回 `set_material_properties(materials, torch.arange(...))` 直接传 torch 张量**同样会失败
   ——PhysX tensor API 要的是 warp 数组 + CPU int32 的 env_ids。

修正（对齐 Isaac Lab 官方 `randomize_rigid_body_material`，见 `envs/mdp/events.py:230`）：
- 顶部新增 `import warp as wp`。
- 读取：`materials = wp.to_torch(physx_view.get_material_properties())` 转成 torch 张量（材质
  buffer 在 CPU 上，故 `materials.device` 变为合法的 cpu 设备，`sample_uniform` 正常）。材质
  shape `(num_envs, num_shapes, 3)`=static/dynamic friction + restitution，`friction_scale`
  `(num_envs,1,1)` 广播缩放前两列（两种摩擦），restitution 不动。
- 写回：`set_material_properties(wp.from_torch(materials.contiguous(), dtype=wp.float32),
  wp.from_torch(env_ids_cpu_int32, dtype=wp.int32))`。

要点/教训：Isaac Lab 里凡是走 `root_physx_view` / `root_view` 的底层 PhysX tensor API
（material、部分 mass/inertia 接口），返回值是 **warp 数组**，须 `wp.to_torch` 进、
`wp.from_torch` 出，且 env_ids 常要求 **CPU int32**；不能像上层 `robot.data.*`/`set_*_index`
那样直接用 torch + `self.device`。改动纯属 DR 实现修复，不影响任何观测/奖励语义。

验证：`train.py --num_envs 64 --max_iterations 2` 冒烟 2 iter 正常跑完，无 traceback/NaN/shape
报错，DR（质量/PD/摩擦）全部生效。
